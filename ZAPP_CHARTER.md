# Zapp Framework

The goal of this project (Zapp) is to create an alternative to Wails (v3alpha.wails.io), Electron, and Tauri. I explored the desired api and functionality in `./context.md`. Always refer to `./context.md` when building out features.

# Goals

We want to create the smallest, fastest, and most flexible webview desktop framework. We prioritize small binaries, fast JS <-> native bridge with minimal/zero-overhead, powerful backend with native C/C++/ObjC interop thanks to Zen-C. I think I have created something with a competitive edge based on the current landscape.

# Current Implementation (3/16/26)

### Shared

- Zen-C `build.zc` and `app.zc` generated by our CLI (along with other template items/dir) leveraging vite templates for frontend
- Additional `@zapp/runtime`, `@zapp/backend`, `@zapp/vite` packages
- `zapp.config.ts` for easy, typed configuration picked up by our CLI
- Optional `backend.ts` to spawn a host compatible (see below) JS context/thread/worker that has some privileged native/backend access
- Dev and build modes, leveraging vite
- build mode options to embed the assets and brotli compress (adds size for performance)
- `build.zc` allows users to link, add additional Zen-C directives, etc
- `app.zc` is the application starting point
- SharedArrayBuffer-like primitive (Sync) that allows the webviews to "share" something in native memory (confirm my understanding)
    - unlocks global backend "store" in native memory, accessible and able to notify/wait
- Event system that allows backend and JS code to emit/listen to events across the application
- "Services" that are native code with optional lifecycle methods that get automatically compiled to TS code that uses the Zapp native bridge to call methods (more efficient than traditional RPC)

### macOS

- WKWebview windows
- optional Zapp WebWorker polyfill (native are still available, no backend/native-bridge):
    - enables JS threads/workers to talk to native bridge (not possible from native WebWorker)
    - single worker/shared worker
    - JavascriptCore by default (no size penalty)
    - Optional QuickJS (~1M size increase)
        - Windows requires this by default so we enable here for flexibility
- `backend.ts` context/"worker" JSC by default (QuickJS opt-in)
- macOS compression framework used to decompress brotli assets
    - we set a custom protocol on WKWebview which means it loses its ability to natively accept brotli (confirm my understanding)
- Sizes (with current features, will grow as we add core functionality) of our example app:
    - `build`: 302K
    - `build --embed-assets`: 385K
    - `build --embed-assets --brotli`: 336K

# Windows

- Webview2 windows
- optional Zapp WebWorker polyfill (native are still available, no backend/native-bridge):
    - enables JS threads/workers to talk to native bridge (not possible from native WebWorker)
    - single worker/shared worker
    - QuickJS as the only default (Webview2/Windows does not come with a JSC counterpart)
- `backend.ts` context/"worker" QuickJS
- No compression lib built-in but we use `https` so the Webview2 will natively accept/decompress
- Sizes (with current features, will grow as we add core functionality) of our example app:
    - `build`: 1.7M
    - `build --embed-assets`: 1.78M
    - `build --embed-assets --brotli`: 1.73M
